# Система управления банковскими картами (Backend)

REST API сервис на Spring Boot, предназначенный для управления банковскими картами, ролевого доступа (USER/ADMIN) и безопасного проведения внутренних переводами между счетами.

## 💡 Технологический стек
* **Язык разработки:** Java 17 / Java 23
* **Фреймворк:** Spring Boot 3.2.5 (Spring Security, Spring Data JPA, Spring Web Validation)
* **База данных:** PostgreSQL
* **Миграции БД:** Liquibase
* **Безопасность:** JWT (JSON Web Token), шифрование номеров карт (JPA Converters / AES), маскирование данных
* **Документация:** Swagger UI / OpenAPI 3.0 (`docs/openapi.yaml`)
* **Тестирование:** JUnit 5, MockMvc, AssertJ, Mockito
* **Оркестрация:** Docker, Docker Compose

---

## 🏗️ Архитектура и ключевые решения

### 1. Безопасность и Персональные Данные
* **Шифрование номеров карт:** Номера карт шифруются перед сохранением в базу данных PostgreSQL с использованием встроенных механизмов (JPA Attribute Converter + AES-256), что исключает утечку данных при компрометации БД.
* **Маскирование:** Полный номер карты никогда не отдается наружу через API для обычных запросов. На уровне DTO (MapStruct) настроено маскирование формата `**** **** **** 1234`.
* **Авторизация:** Безопасность реализована на базе JWT-токенов. После авторизации (`/api/v1/auth/signin`) токен парсится кастомным `JwtTokenFilter`, а идентификатор пользователя `userId` внедряется в контекст запроса.

### 2. Управление Ролями
* **ADMIN:** Имеет полный CRUD-доступ к картам всех пользователей, может блокировать/активировать карты, обрабатывать входящие заявки на блокировку и управлять учетными записями пользователей.
* **USER:** Имеет доступ только к своим картам. Может просматривать список карт (с пагинацией и фильтрацией), запрашивать блокировку и совершать переводы.
* *Валидация бизнес-логики:* При переводах между картами строго проверяется, что обе карты принадлежат именно авторизованному в данный момент пользователю.

## 📁 Структура проекта
Проект организован по классической многослойной архитектуре (Layered Architecture):
* `config/` — Конфигурационные классы (Security, OpenAPI/Swagger).
* `controller/` — REST-контроллеры, валидация входящих DTO, интеграция со Swagger.
* `dto/` — Объекты передачи данных (входящие/исходящие контракты API).
* `exception/` — Исключения и обработчик (NotFoundException, ForbiddenException и тд).
* `entity/` — Сущности базы данных (User, Card, BlockRequest).
* `repository/` — Репозитории Spring Data JPA для работы с БД.
* `service/` — Классы бизнес-логики приложения.
* `util/` — Утилитарные классы, конвертеры и перечисления (Enums).

## 🚀 Инструкция по запуску

### Предварительные требования
Убедитесь, что у вас установлены:
* Docker и Docker Compose
* Maven (для локальной сборки)

### Локальный запуск в Docker-окружении

1. Клонируйте репозиторий:
   ```bash
   git clone [https://github.com/Dj474/Bank_test.git](https://github.com/Dj474/Bank_test.git)
   cd bank_rest
   ```
2. Соберите проект и запустите инфраструктуру (БД + Сервис) одной командой:
    ```bash
    docker-compose up --build -d
    ```
> Liquibase автоматически применит все миграции из директории `src/main/resources/db/migration` при старте контейнера базы данных.

⚠️ **Важно:**

В корне проекта уже подготовлен и добавлен в Git файл `.env` с тестовыми секретами и настройками подключения.

> В коммерческой разработке данный файл является конфиденциальным и обязательно заносится в `.gitignore`. В рамках данного тестового задания он оставлен в репозитории исключительно для удобства проверяющего, чтобы проект запускался «из коробки» без необходимости ручной настройки переменных окружения.

Приложение будет доступно по адресу: [http://localhost:8080](http://localhost:8080)

## 📄 Документация API

### 📊 Документация OpenAPI и Swagger UI

* **Локальный файл спецификации:** Находится по пути `docs/openapi.yaml`. Он генерируется программно внутри интеграционного теста `OpenApiGeneratorTest`. Файл автоматически перезаписывается актуальными данными при каждом запуске тестов (`mvn test`) или полной сборке проекта (`mvn clean verify`). Также его можно обновить вручную, запустив этот тест напрямую из IDE.
* **Интерактивный UI (Swagger UI):** Когда приложение запущено локально, вы можете протестировать все эндпоинты в интерактивном режиме по адресу:  
  [http://localhost:8080/swagger-ui.html](http://localhost:8080/swagger-ui.html)

*Для тестирования защищенных методов используйте кнопку **Authorize** в Swagger UI, вставив туда JWT Access Token, полученный из эндпоинтов регистрации или авторизации.*
